## 最優先

### P1. README・GitHub Pagesの価値訴求不足

- **問題**: README冒頭・GitHub Pagesタイトルが「Raspberry Shakeモニタ」に見え、月次レポート・HVSR・地盤解析・発震機構解という独自性が伝わらない（現状評価レポート10節）。
- **改善内容**: フェーズ3の情報設計案に基づきREADME・`docs/index.html`を再構成する。
- **利用者への効果**: 初見の訪問者がプロジェクトの独自性を理解できる。
- **実装コスト**: 中（文章構成の見直し、コード変更なし）。
- **破壊的変更**: なし（説明文の変更のみ）。
- **対象ファイル**: `README.md`、`docs/index.html`。
- **依存関係**: フェーズ3の成果物。
- **テスト方法**: `test_template_parity.py`への影響なし（Jinja2テンプレートではない）。目視確認、リンク切れチェック。
- **本ロードマップでの扱い**: **フェーズ6で実装対象**（ユーザー承認済み）。

### P2. CI/CDの不在

- **問題**: `.github/workflows/`が空。既存テスト（110件以上）が手動実行のみで、コミット・プッシュ時に自動検証されない。
- **改善内容**: GitHub Actionsで`push`/`pull_request`時に`pytest src/verify_filter.py src/test_api_events.py src/test_hvsr_weekly.py src/test_api_hvsr_history.py src/test_city_matching.py`と`python src/test_template_parity.py`を実行するワークフローを追加する。
- **利用者への効果**: リグレッションの早期発見。OSSとしての信頼性向上（外部貢献者がPRを出しやすくなる）。
- **実装コスト**: 低〜中（`geopandas`等の重量級依存のCI環境構築が必要、iMac固有の環境変数・SeedLink接続に依存するテストは除外要）。
- **破壊的変更**: なし。
- **対象ファイル**: `.github/workflows/test.yml`（新規）。
- **依存関係**: `requirements.txt`の整備（現状ルート直下に存在しない。README記載のpipコマンドと`requirements.lock.txt`の関係整理が前提）。
- **テスト方法**: ワークフロー自体をpush・PRで実行し、既存110件超のテストが緑になることを確認。

### P3. 月次レポートのエグゼクティブサマリー・データ品質表示の不足

- **問題**: 「解説・総評」は詳細だが、検出率・誤検出候補・欠測時間・データ品質・前月比較・HVSR推移という定量的自己評価が欠けている（現状評価レポート5節）。
- **改善内容**: フェーズ4の月次レポート改善プラン（別紙）を参照。
- **利用者への効果**: 「観測点として何が分かったか」が定量的に伝わる。
- **実装コスト**: 中〜高（`monthly_report.py`のロジック追加、既存レポート生成物との整合確認）。
- **破壊的変更**: 既存レポートのHTML構造が変わるため、`test_template_parity.py`相当の回帰確認が必要（月次レポートには現状専用テストがない）。
- **対象ファイル**: `src/monthly_report.py`。
- **依存関係**: `trigger_log.jsonl`（既存）、`data/hvsr_history.jsonl`（新設、まだ1週分のみ）。
- **テスト方法**: 過去月データでの生成テスト（fixtureまたは既存キャッシュ利用）、生成HTMLの構造検証テストの新規作成。
- **本ロードマップでの扱い**: プランのみ。実装はユーザー判断待ち。

## 重要

### I1. 誤検出候補の明示（イベント分類の第一歩）

- **問題**: 自局トリガの99%が「全国地震と時刻一致しない」もので、その内訳（生活振動・交通振動・センサーノイズ等）が不明。
- **改善内容**: フェーズ4のイベント分類案（ルールベース・スコアリング方式）の一部として、STA/LTA・継続時間・三成分比等の特徴量による粗い分類を月報・WebUIに表示する。
- **利用者への効果**: 「検出はしているが地震ではない」ことが分かり、システムの限界が正しく伝わる。
- **実装コスト**: 中〜高（特徴量設計・閾値調整に検証が必要）。
- **破壊的変更**: なし（表示追加のみ、検出ロジック自体は変更しない）。
- **対象ファイル**: `src/jma_intensity_tui.py`（特徴量算出を追加する場合）、`src/monthly_report.py`（表示）。
- **依存関係**: `trigger_log.jsonl`のスキーマ拡張が必要な可能性。
- **テスト方法**: 既知の交通振動・地震イベントのラベル付きデータでの分類精度検証（fixtureで実施、実データ検証は別途）。

### I2. HVSRの表示拡充（卓越周期・信頼度・前週比）

- **問題**: 現状のWebUIパネルは卓越周波数・H/V曲線・SESAME簡易バッジのみ。卓越周期・ピーク幅・信頼度の言語化・前週比・長期トレンドがない。
- **改善内容**: フェーズ5のHVSR改善プランを参照。
- **利用者への効果**: グラフの読み方が分からない人でも意味を理解できる。
- **実装コスト**: 中（`hvsr_weekly.py`の出力フィールド追加＋`dashboard.html`の表示追加）。
- **破壊的変更**: `data/hvsr_history.jsonl`のスキーマに追加フィールドを加える場合、後方互換性（既存1件のレコードにフィールドがない）を考慮する必要がある。
- **対象ファイル**: `src/hvsr_weekly.py`、`src/jma_intensity_web.py`、`src/templates/dashboard.html`。
- **依存関係**: 複数週分のデータ蓄積後でないと「前週比・トレンド」は意味を持たない。
- **テスト方法**: `test_hvsr_weekly.py`への追加テスト、合成データでの表示確認。

### I3. weather_noteの実質化

- **問題**: `hvsr_weekly.py`の`weather_note`フィールドが常に空文字。降雨・強風の影響評価ができない。
- **改善内容**: 気象データ（Weathercloud富士見台等、既存メモリ参照）または風雨に敏感な帯域パワー（既存`_compute_band_powers`の15-44Hz帯）との突合で自動記入する。
- **利用者への効果**: HVSR結果の解釈時に「悪天候の影響」を切り分けられる。
- **実装コスト**: 中（外部気象APIまたは既存帯域パワーとの相関ロジック）。
- **破壊的変更**: なし（フィールドの中身が変わるだけ）。
- **対象ファイル**: `src/hvsr_weekly.py`。
- **依存関係**: 外部気象データソースの選定（新規）。
- **テスト方法**: 既知の悪天候日・平常日での比較テスト。

### I4. POST /api/analyzeのdurationバリデーション

- **問題**: `duration`パラメータに上限チェックがない。
- **改善内容**: `_CONFIG_LIMITS`同様の範囲チェックを追加（例: 60〜1800秒）。
- **利用者への効果**: 直接の利用者体験には影響しないが、堅牢性向上。
- **実装コスト**: 低。
- **破壊的変更**: なし（既存の妥当な範囲の呼び出しには影響しない）。
- **対象ファイル**: `src/jma_intensity_web.py`（`api_analyze_start`関数）。
- **依存関係**: なし。
- **テスト方法**: `test_api_events.py`と同様のパターンで境界値テストを追加。

- **実装コスト**: 中。
- **破壊的変更**: なし。
- **対象ファイル**: `src/monthly_report.py`。
- **依存関係**: なし。
- **テスト方法**: 既知の停電期間（2026-06-03台風6号）のデータでの検証。

### M3. requirements.txtの整備

- **問題**: ルートに`requirements.txt`がなく、READMEのpipコマンドと実際の依存関係の対応が不明瞭。`requirements.lock.txt`という未追跡ファイルが別途存在している。
- **改善内容**: `requirements.txt`（緩いバージョン指定）と`requirements-lock.txt`（開発機のfreeze結果）を整理し、READMEのインストール手順を`pip install -r requirements.txt`に統一する。
- **利用者への効果**: 新規セットアップの再現性向上。
- **実装コスト**: 低。
- **破壊的変更**: なし。
- **対象ファイル**: `requirements.txt`（新規）、`README.md`。
- **依存関係**: なし。
- **テスト方法**: クリーンな仮想環境での`pip install`検証。

## 実装順序の推奨（ユーザー判断用の参考）

このロードマップ自体はプランであり実装しないが、仮に着手する場合の依存関係を踏まえた順序の目安を示す。

1. P2（CI/CD）→ 以降の全ての変更を安全に検証する基盤になるため最初に着手する価値が高い
2. M3（requirements整備）→ P2のCI構築の前提
3. I4（durationバリデーション）→ 低コスト・低リスクで独立して実施可能
4. P3・I1（月報改善・誤検出候補）→ 相互に関連するため合わせて設計
5. I2・I3（HVSR拡充）→ データ蓄積が進んでから着手するとより意味のある表示になる
6. M1・M2 → ドキュメント作業と並行可能
7. F1〜F3 → 上記が安定してから検討
