# HVSR改善プラン

- 作成日: 2026-07-14
- 前提: `src/hvsr_weekly.py`実装の確認、および2026-07-14のiMac本番デプロイ時に得た実データ（初回実行、`status=ok`だが`amplitude_ok=false`/`stability_ok=false`、H/V比が0.2〜20Hzでほぼ平坦1.2〜1.6）に基づく
- 位置づけ: **プランのみ**。実装はユーザー判断待ち。HVSRからVs30や地盤種別を断定しない方針を維持する。

## 実装済み内容の評価

| 観点 | 現状実装 | 評価 |
|---|---|---|
| 常時微動解析としての妥当性 | 深夜02:00〜05:00 JST（3時間）を対象時間帯とする設計。人為ノイズが少ない時間帯選定は妥当 | 妥当。ただし初回実データで棄却率56%と想定より高く、深夜でも交通・機器ノイズの影響が大きい可能性がある |
| 地震イベントの除外 | STA/LTAアンチトリガ（`[0.5, 2.0]`範囲外を含む窓を棄却、`is_window_valid`）。3成分すべてで判定し、いずれか1つでも棄却対象なら窓全体を棄却する設計（`compute_hvsr_from_traces:403-405`） | 妥当。ただし棄却閾値`[0.5, 2.0]`が固定値で、季節・時間帯による背景ノイズレベルの変動を考慮していない |
| 窓長 | 40秒 | SESAMEの「目的周波数の逆数の10倍以上」の推奨に対し、0.2Hz以下の低周波を見る場合は窓長不足の可能性がある（0.2Hzの10倍則では50秒必要）。ただし設計書では0.83Hz試算を基準に汎用性重視で40秒を選択したと明記されており、トレードオフとして妥当な判断 |
| オーバーラップ | 50% | SESAME推奨範囲内。妥当 |
| テーパー | 5%コサイン（Tukey窓、`apply_cosine_taper`） | SESAME推奨の範囲内（5〜10%が一般的）。妥当 |
| 平滑化 | Konno-Ohmachi（b=40、`normalize=True`） | 実装時に発見・修正されたバグ（`normalize=False`でナイキスト近傍が不当に増幅される問題）が解消済み。SESAME標準・Geopsy実装との整合を確認済みで信頼できる |
| 水平成分の合成方法 | 幾何平均（`h_amp = sqrt(amp_n * amp_e)`） | Nakamura法の一般的な合成法の一つとして妥当。SESAMEは二乗平均平方根（RMS）合成も許容しており、幾何平均・RMSどちらでも大差ないとされるため問題なし |
| 外れ値除去 | STA/LTAアンチトリガのみ。窓ごとのピーク周波数分布に対する統計的外れ値除去（IQR法等）は未実装 | 弱点。1本の異常な窓が対数平均スタッキングに与える影響は限定的（対数平均は外れ値に比較的頑健）だが、明示的な外れ値除去は行われていない |
| 時間帯依存 | 深夜固定（02:00-05:00 JST）のみ。昼夜比較機能は`microseism.py`に既存だが`hvsr_weekly.py`とは連携していない | 弱点。時間帯によるH/V比の変動を捉える仕組みがない |
| 風雨・交通ノイズの影響 | `weather_note`フィールドが常に空文字（未実装、改善ロードマップI3で既述） | 明確な弱点 |
| ピークの安定性 | `peak_freq_per_window`（各窓の未平滑化生データでの個別ピーク）の標準偏差を算出し、SESAME Table 3の帯域別閾値と比較（`stability_ok`） | 実装はされているが、**母集団の不整合**がある。`peak_freq_per_window`は未平滑化・窓別の生ピークである一方、最終`peak_frequency_hz`はスタッキング後・平滑化後の曲線から算出されており、両者は異なる統計量に基づく。この設計自体は「参考記録」としての位置づけであれば許容範囲だが、解釈の際は注意が必要 |
| 信頼区間 | なし（SESAME原典の`σA(f0)`等は未実装、設計書でも意図的に不採用と明記） | 意図的な簡略化。将来的にクライテリアを追加するかはコスト対効果次第 |
| SESAME基準への適合可能性 | 9項目中3項目（`window_length_ok`/`amplitude_ok`/`stability_ok`）のみ選択的実装。正式な「reliable curve」「clear peak」判定ではないと明記済み | 設計判断として妥当。ユーザーの「安易に断定しない」方針とも整合 |

## 初回実データが示す課題

2026-07-14のiMac本番投入時に実行した初回バッチ（2026-07-13深夜データ）は、有効窓237/539（棄却率56%）で`status=ok`だったが、H/V比曲線が0.2〜20Hzの全帯域でほぼ平坦（1.2〜1.6程度）、ピークがナイキスト近傍（15.89Hz）に出た。SESAME簡易基準は`amplitude_ok=false`（振幅1.62<2.0）・`stability_ok=false`（標準偏差13.79Hzと非常に大きい）で、これは実装のバグではなく「この夜の常時微動には明確な卓越周波数が検出できなかった」ことを正しく反映している可能性が高い。

ただし、この1回の結果だけでは以下の3つの可能性を区別できない。

1. 観測点（R38DC、箱根西麓ローム層）の地盤特性として、実際に明確な単一卓越周波数を持たない（複数層構造・インピーダンスコントラストが弱い等）
2. その晩のノイズ条件（棄却率56%という高さが示唆する通り）が悪く、たまたま不安定な結果になった
3. `peak_frequency_from_curve()`の単純な`argmax`実装が、ノイズフロアの高周波側単調増加を誤ってピークとして拾っている

この切り分けには、複数週分のデータ蓄積（最低でも4〜8週）を待つ必要がある。**現時点でこの1回のデータから地盤特性を結論づけるべきではない**。

## Vs30・地盤種別の断定に関する方針（重要）

ユーザーの明示的な制約「HVSRからVs30や地盤種別を安易に断定しない」「推定根拠が弱い場合は表示しない」に基づき、以下を推奨する。

- 現状の実装・表示は卓越周波数・H/V比という直接観測量に留めており、Vs30や地盤種別への変換は一切行っていない。この方針を維持する。
- 仮に将来Vs30推定（経験式によるVs30-卓越周波数の関係式等）を検討する場合も、単一手法・単一観測点からの推定である旨、地盤種別の判定に用いる場合は複数の経験式間で結果が大きく異なりうる旨を併記すべきである。
- 現時点（データ1週分のみ、かつそのデータが不安定）では、地盤種別に関するいかなる言及も時期尚早であり、実装しないことを推奨する。

## 改善提案（優先順位付き、プランのみ）

1. **低コスト・低リスク**: 卓越周期・除外窓数の表示追加（単純な追加計算のみ）
2. **中コスト**: SESAME簡易バッジ3項目の統合「信頼度」表示（参考指標であることを明記）
3. **中コスト**: `weather_note`の実質化（改善ロードマップI3、既存帯域パワーとの相関または外部気象データ）
4. **中コスト**: ピーク幅（半値幅）の算出、ピークが不明瞭な場合の扱い設計
5. **データ蓄積待ち**: 前週比・前月比・長期トレンド（4〜8週分のデータが溜まってから着手するのが合理的）
6. **要検討**: `peak_freq_per_window`（安定性クライテリア用）と最終`peak_frequency_hz`（スタッキング後）の母集団不整合について、次回の設計見直し時に整理する（緊急性は低いが、ドキュメント上「参考記録に過ぎない」ことをより明確にする価値はある）

いずれも実装はユーザー判断待ちとする。
