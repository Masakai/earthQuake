# 月次レポート改善プラン

- 作成日: 2026-07-14
- 前提: [現状評価レポート](2026-07-14-project-wide-assessment.md)の訂正済み記述（`generate_commentary()`は完全自動生成のテンプレート文、「解説・総評」は`load_manual_commentary()`による手動執筆ファイル読み込み）に基づく
- 位置づけ: **プランのみ**。実装はユーザー判断待ち。

## 現状の正確な構造（再確認）

`monthly_report.py`のレポート組み立ては2種類の異なる仕組みが混在している。

| セクション | 生成方法 | 関数/仕組み |
|---|---|---|
| 全体像／注目イベント／地域別の活動傾向／総括 | 完全自動生成（統計値→テンプレート文への条件分岐） | `generate_commentary()` |
| 自局トピック | 自動集計＋テンプレート文（地盤特性の恒久注記は固定文） | `compute_station_topics()` / `make_station_topics_html()` |
| 解説・総評 | **手動執筆**（`data/monthly_report/commentary_YYYYMM.html`を読み込み） | `load_manual_commentary()` |

ユーザー指摘の「事実の列挙に寄りやすい」は主に自動生成部分（`generate_commentary()`）に該当する。手動執筆部分（解説・総評）は既に詳細だが、これは自動化されていないため毎月の執筆負担がかかっている。

## 必須候補の評価

| 項目 | 現状 | 改善案 |
|---|---|---|
| エグゼクティブサマリー | なし（総括はあるが定型文） | レポート冒頭に自動生成の数値サマリーカード（総地震数・最大震度・自局検出率・データ品質フラグ）を追加。手動執筆の「解説・総評」より前に配置し、詳細を読まなくても要点が分かる導線にする |
| 今月の注目地震 | あり（`_notable_event_comment`によるM5以上のコメント） | 現状維持。前月の注目地震との比較コメントを追加余地あり |
| 自局最大イベント | あり（`compute_station_topics`の`max_I`） | 現状維持で十分 |
| 公式震度との差 | 部分的（`local_intensity_badge`で自局震度は表示するが、公式震度との差分値は非表示） | 自局検出済み地震について、P2P情報に含まれる公式最大震度と自局震度の差分を明示的に算出・表示する |
| 検出率 | データはあるが「検出率」という指標としては未算出 | `len(detected_quakes) / len(quakes)`（有感地震のうち自局で波形検出できた割合）を明示的な指標として算出・表示する |
| 誤検出候補 | なし | `topics['total']`（自局トリガ総数）から`detected_quakes`件数を引いた残数を「未特定検出（地震以外の可能性を含む）」として明示する。個々の分類は将来のイベント分類機能（下記）に委ねる |
| 欠測時間 | なし | `bp_history.jsonl`等のタイムスタンプ間隔から一定閾値（例: 10分以上のギャップ）を欠測区間として検出し、月内の欠測合計時間を表示する |
| データ品質 | なし | 欠測時間・トリガ総数の異常値（前月比で極端に多い/少ない）をフラグとして表示する |
| 前月比較 | なし | 前月のレポートJSON（統計値を別途保存する仕組みが必要）と比較し、地震数・自局検出率等の前月比を表示する |
| 地盤特性の変化 | なし（地盤特性自体は固定文として恒久表示） | HVSRデータが複数週分蓄積された後、月内のピーク周波数の変動幅を要約表示する |
| HVSRの推移 | なし（月報とHVSR週次データの連携が未実装） | `data/hvsr_history.jsonl`から該当月の週次データを抽出し、月次レポートに小さなグラフまたは表として掲載する |
| 自動考察と事実の明確な分離 | 部分的（免責文一行のみ） | 自動生成セクションには「※自動集計」、手動執筆セクションには「※観測者による考察」のラベルを明示的に付与する |
| 「推定」「可能性」「未確認」の表示 | 一部使用されている（既存の地盤注記等） | 誤検出候補・データ品質フラグ等、新規追加する自動判定項目には必ずこれらの語を用い、断定表現を避ける |

## イベント分類（ルールベース・スコアリング方式）

ユーザー要求により機械学習には進まず、説明可能なルールベースまたはスコアリング方式を優先する。

### 分類カテゴリ（案）

- 地震候補
- 生活振動候補
- 交通振動候候補
- 設備振動候補
- センサー異常候補
- 未分類

### 利用可能な特徴量の評価

| 特徴量 | 現状のデータ取得状況 | 分類への使いやすさ |
|---|---|---|
| 継続時間 | `trigger_log.jsonl`に記録があるか要確認（現状未確認、`compute_stalta`の出力からは算出可能） | 交通振動（短時間・単発）と地震（長時間・複数相）の区別に有効 |
| STA/LTA | 既存（`ratio`フィールド） | 既にある。閾値だけでは地震/非地震を区別できないため他特徴量と組み合わせる必要 |
| 三成分比 | ENZ/ENN/ENE個別の振幅は取得可能だが、トリガ記録に三成分比として保存されていない | 追加のデータ収集・記録項目の拡張が必要 |
| 卓越周波数 | 個別イベントごとの算出は未実装（HVSR週次計算とは別に、イベントごとのFFTが必要） | 交通振動（高周波優勢）と地震（低周波〜広帯域）の区別に有効。実装コストは中程度 |
| 高周波成分比 | 既存の帯域パワー計算（`_compute_band_powers`、4帯域）の枠組みを流用可能 | 実装しやすい。交通・豪雨帯域（15-44Hz）が優勢なら地震以外の可能性が高いと推定できる |
| P波・S波らしい位相差 | 未実装 | 実装コストが高い。単一観測点では信頼度が低い可能性がある |
| 外部地震情報との時刻一致 | 既存（`load_trigger_hhmm_set`の仕組みがまさにこれ） | 既にある。「地震候補」の一次判定として使える |
| 震源距離 | 外部地震情報と一致した場合のみ算出可能（一致しない場合は震源不明） | 一致した場合のみ有効 |
| 時間帯 | トリガ記録のタイムスタンプから算出可能 | 生活振動・交通振動の時間帯パターン（日中に多い等）との相関に使える |
| 過去イベントとの類似度 | 未実装（特徴量ベクトルの定義が前提） | 実装コストが高く、後回しが妥当 |

### 優先実装順（提案、プランのみ）

1. 外部地震情報との時刻一致（既存）で「地震候補」を一次分類
2. 高周波成分比（既存の帯域パワー枠組み流用）で「交通振動候補」「設備振動候補」を粗く分離
3. 時間帯パターンで生活振動候補の補助判定
4. 継続時間・卓越周波数は次段階（実装コスト中）
5. 三成分比・位相差・類似度は将来構想（実装コスト高、単一観測点での信頼性検証が必要）

各特徴量にスコアを与え、閾値を超えたカテゴリに分類するスコアリング方式を推奨する（単一の決定木的ルールだと境界条件で誤分類が増えやすいため）。ただし、この設計の詳細（各特徴量の重み・閾値）は実データでの検証を要するため、本プランでは方向性のみを示す。

## 破壊的変更の考慮

`generate_commentary()`・`compute_station_topics()`の出力HTMLを変更する場合、既存の月次レポート（過去分含む）とのビジュアル整合性を壊さないよう、新規セクションは既存セクションの前後に追加する形とし、既存セクションの文言・構造自体は変更しない方針を推奨する。

## テスト方法（提案）

- `monthly_report.py`には現状専用のテストが存在しない。改善実装時は`test_monthly_report.py`を新設し、fixtureデータ（合成P2P地震リスト・合成`trigger_log.jsonl`）で検出率・欠測時間の計算ロジックを単体テストする。
- 生成HTMLの構造検証（新規セクションの見出しタグの存在確認等）を`test_template_parity.py`に類する形で追加する。
