# 設計書: 地図の市区町村塗りつぶし照合バグ修正

- 作成日: 2026-06-27
- 対象プロジェクト: earthQuake（地震計測震度ダッシュボード）
- 要件トレーサビリティ: 地図塗りつぶし不一致バグ（全国 618/4372 局・約14% が塗り漏れ）
- 関連Issue / PR: なし（口頭要件）

## 背景・目的

地図上で観測点ごとの震度バッジ（円形の数字）は座標ベースで正しく表示されるが、市区町村ポリゴンの塗りつぶしが行われないケースが全国で 618/4372 観測点（約14%）発生している。スクリーンショットでは静岡県の浜松市・森町が白いまま、バッジだけ浮いた状態になる。

原因は `src/templates/dashboard.html` の `extractCity(addr)` が、気象庁観測点マスター（`data/jma_stations.json`）の `addr` 文字列を市区町村名へ正しく変換できないこと。具体的には以下の調査で確定済み（本設計書の数値はすべて `.venv/bin/python` で `data/jma_stations.json` と `data/geojson/` を実走査して算出）。

- 旧ロジックの塗り漏れ実測値: **618/4372**（問題報告値と一致）

### 確認した事実（調査ログ）

1. **`extractCity` の使用箇所は `drawCityLayer` 内の1箇所のみ**（`dashboard.html:1119`）。定義は `dashboard.html:1187`。バッジ描画 `drawBadgeLayer`（1076行〜）は `_stationMaster[p.addr]` の lat/lon を直接使う座標ベースで、`extractCity` に依存しない。したがって本修正のバッジへの影響はない。
2. **GeoJSON 構造**: `data/geojson/{pref2桁}/{市区町村コード}.json`。各 feature は `properties` に `N03_001`(県)・`N03_004`(市区町村名)・`N03_005`(政令市の区名)・`N03_007`(コード) を持つ。geometry は `Polygon`（座標ネスト深さ3）と `MultiPolygon` が混在。政令市は区ごとにファイルが分かれ、`N03_004` は政令市名（例: 22138/22139/22140.json はいずれも `N03_004='浜松市'`）、区名は `N03_005`（`中央区`/`浜名区`/`天竜区`）に入る。**現行の塗り照合は `N03_004` のみ使用**（区を区別しない＝政令市は市全体が同色）。
3. **全国規模**: 全 GeoJSON で feature 124,132 件・頂点 15,394,934 点。静岡県単独でも 1,828 feature、葵区1ファイルで13,880頂点。
4. **観測点座標精度**: lat の小数桁分布は {2桁: 3905局, 1桁: 467局}。小数1桁は緯度方向で約11kmの誤差を含む。
5. ライブラリ: サーバ側 `.venv` に shapely 2.1.2 / geopandas 1.1.3 が導入済み。

### 不一致の主因（調査で確定）

| 分類 | 例 (addr → 旧extractCity結果) | GeoJSON側 | 件数規模 |
|------|------|------|------|
| 政令市の区 | `浜松中央区高丘東` → `浜松中央区` | `N03_004=浜松市` | 最大要因。全国20政令市 |
| 県内プレフィックス | `静岡森町森` → `静岡森町` / `静岡菊川市赤土` → `静岡菊川市` | `森町` / `菊川市` | 一部県 |
| 地方名プレフィックス | `上川地方上川町越路` → `上川地方上川町` | `上川町` | 北海道 |
| 最短マッチの取りこぼし | `四日市市…` → `四日`+`市`で誤停止 / `大町町` → `大町` | `四日市市` / `大町町` | 数件 |
| 異体字 | `梼原町広野`（梼 U+68BC） | `檮原町`（檮 U+6AAE） | 1町2局 |
| 空港等（市町村外） | `新千歳空港` `仙台空港` | 該当なし | 13局。塗り対象外＝正常 |

## 設計概要

**案B（`extractCity` のルール修正）を採用する。** クライアント側 `dashboard.html` 内で完結する文字列正規化の強化のみで、サーバ変更・新規データファイル・新規ライブラリを伴わない。

正規化を3段階の照合に置き換える。照合先は「その県の GeoJSON が実際に持つ市区町村名集合（`N03_004`）」と「区名→政令市名の逆引き（`N03_005`→`N03_004`）」。これらは `fetchCityGeoJson` で取得済みの GeoJSON feature から構築できるため、追加のネットワーク取得は不要。

1. **政令市プレフィックス照合**: addr が20政令市のいずれかの「市を除いた名」で始まり、続きが `X区` で、`X区` がその県の `N03_005` 集合にあれば、対応する `N03_004`（政令市名）を採用。
2. **最長前方一致**: addr の先頭から、その県の既知市区町村名（`N03_004`）に最長で前方一致するものを採用。最短マッチではなく最長一致にすることで `四日市市`・`大町町` を救済。
3. **プレフィックス除去後の最長前方一致**: 先頭1〜4文字を削った部分文字列に対し最長前方一致を試行。`静岡森町`・`上川地方上川町` などの県名／地方名プレフィックスを救済。
4. **異体字正規化テーブル**: 照合前に addr へ最小限の異体字置換を適用（`梼`→`檮` の1件のみ確認済み）。

| 方式 | 塗り漏れ |
|------|---------|
| 旧ロジック | 618 / 4372 |
| 案B（区マッピング＋プレフィックス除去・最短マッチ） | 31 / 4372 |
| 案B改（区マッピング＋最長前方一致＋プレフィックス除去） | 15 / 4372 |

残る13件は `新千歳空港`・`仙台空港`・`成田国際空港` 等、市町村ポリゴンを持たない施設で、塗り対象外。したがって**塗り対象の全局で塗り漏れ0を達成**。

## 検討した選択肢と却下理由

### 案A: 点-ポリゴン判定（point-in-polygon）

観測点 lat/lon が GeoJSON のどの市区町村ポリゴンに含まれるかで判定し、addr 文字列解析を全廃する案。**却下**。

却下理由（いずれも調査で裏取り済み）:

1. **座標精度が不足**: 観測点座標は小数1〜2桁（467局が小数1桁＝約11km誤差、3905局が小数2桁＝約1.1km誤差）。市区町村境界付近・小面積の区・離島で誤判定が頻発する。誤判定は「白いまま」より悪く、**隣接市を誤った震度色で塗る**ため、誤情報になる。本プロジェクトの方針（[[feedback_detection_policy]]: 見逃しより誤りが深刻な領域では誤りを避ける）に反する。
2. **クライアント実装コストと負荷**: Leaflet 単体に point-in-polygon はなく Turf.js 等の追加ライブラリが必要。地図描画は地震選択（クリック/WSプッシュ）と `zoomend`（`dashboard.html:1198`）で都度走る。MultiPolygon・飛び地・1.4万頂点級ポリゴンに対し、その県の観測点数十局 × 数百〜千ポリゴンの内外判定を毎回行うのは、文字列照合（ハッシュ＋前方一致）と比べ桁違いに重い。
3. **サーバ事前計算案（観測点→市区町村コード対応表をビルド時に1度作る）も却下**: shapely で対応表自体は生成可能だが、(1) の座標精度問題は解消せず誤割当が残る。さらに新規データファイル（対応表 JSON）の生成・配信・市町村合併時の再生成という運用負担と、コードとデータの乖離リスク（CLAUDE.md グローバル方針）を新たに抱える。案Bは追加データファイル・追加取得なしで塗り対象0達成のため、事前計算の利点が乏しい。

案Aが優位になるのは、観測点座標が高精度（小数4桁以上）で、かつ addr 表記が照合不能なほど不規則な場合。本データはいずれにも当たらない。

### データフロー（変更後）

```
drawCityLayer(quake)
  └ prefsNeeded ごとに fetchCityGeoJson(code)  ← 既存。変更なし
```

- **セキュリティ**: クライアント JS の文字列処理のみ。外部入力経路・パストラバーサル等の新規リスクなし。`/api/geojson` のサーバ実装は変更しない。

## 区単位対応（確定アルゴリズム・全4372局で実測検証済み）

ユーザー回答により、政令市を区単位で塗り分ける。照合ロジックを以下に確定し、`.venv/bin/python` で全4372局シミュレーション済み（**塗り対象の塗り漏れ0、残13件はすべて空港**）。

### 照合キーの定義

- 区を持つ政令市: `pref + '::' + N03_004 + '::' + N03_005`（例 `静岡県::浜松市::中央区`）
- それ以外の市町村: `pref + '::' + N03_004`（区なし）

`cityScale`（キー→最大震度）と GeoJSON style/onEachFeature の両方でこの同一キー規則を使う。style 側は `feature.properties.N03_005` があれば区付きキー、なければ区なしキーを作る。

### resolveCity(addr, pref, keys) アルゴリズム

`keys` = その県の GeoJSON から構築した `(N03_004, N03_005|null)` タプル集合。`cities_with_ku` = 区を持つ政令市名集合、`plain` = 区なし市町村名集合。

1. 異体字正規化（`梼`→`檮`）を addr に適用。
2. **政令市＋区照合**: addr の先頭、および先頭1〜2文字を除去した部分文字列について、20政令市の「市あり名/市なし名」で始まり直後が `〜区` で、`(政令市名, 区名)` が `keys` にあれば採用。先頭除去は `大阪堺市堺区…`（県名プレフィックス＋政令市の二重構造）を救済するため。長いヘッド優先でソート。
3. **区なし市町村の最長前方一致**: addr の先頭、および先頭1〜4文字を除去した部分文字列について、`plain` に最長前方一致する市町村名を採用。`静岡森町`→`森町`、`四日市市`（最長一致で救済）、`上川地方上川町`→`上川町` に対応。
4. いずれも一致なしは `null`（塗らない＝空港等）。

### 区単位での影響範囲の差分（市単位案からの変更点）

- `cityScale` キーを区付き（政令市のみ）に拡張。`resolveCity` は `(N03_004, N03_005)` を返す。
- `drawCityLayer` の `Promise.all` 後に構築するのは `cityNames`（pref→Set(N03_004)）ではなく **`keys`（pref→Set((N03_004,N03_005)タプル)）**。
- style/onEachFeature のキー生成を `feature.properties.N03_004 (+ '::' + N03_005 if 区)` に変更（**これは市単位案では不要だった変更点**）。これにより政令市の各区が独立に塗られる。
- 非政令市の見た目・キーは現行どおり（区なし）。
